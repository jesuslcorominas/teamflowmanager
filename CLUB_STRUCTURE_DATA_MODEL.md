# Club Structure Data Model - Design Document

## Overview

This document describes the data model design for the Club structure in TeamFlow Manager. The Club structure establishes a hierarchy for managing multiple teams under a single organization (club) with defined membership and roles.

## Collections

### 1. clubs Collection

The `clubs` collection represents the top-level organizations in the system. Each club is owned by a user and can have multiple members with different roles.

#### Collection Path
```
/clubs/{clubId}
```

#### Document Structure

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | String | Yes (auto) | Firestore document ID, automatically assigned |
| `ownerId` | String | Yes | UID of the user who owns/created the club |
| `name` | String | Yes | Display name of the club |
| `invitationCode` | String | Yes | Unique code for inviting members to join |

#### Example Documents

**Example 1: Football Club**
```json
{
  "id": "club_abc123xyz",
  "ownerId": "user_firebase_uid_001",
  "name": "Club Ejemplo",
  "invitationCode": "INVITE123"
}
```

**Example 2: Youth Sports Academy**
```json
{
  "id": "club_def456uvw",
  "ownerId": "user_firebase_uid_002",
  "name": "Academia Deportiva Juvenil",
  "invitationCode": "ACAD2024"
}
```

**Example 3: Community Sports Club**
```json
{
  "id": "club_ghi789rst",
  "ownerId": "user_firebase_uid_003",
  "name": "Club Deportivo Municipal",
  "invitationCode": "CDM2025"
}
```

#### Business Rules

1. **Owner Assignment**: The `ownerId` field must be set to the authenticated user's UID when creating a club
2. **Unique Invitation Codes**: Each `invitationCode` should be unique across all clubs to prevent conflicts
3. **Name Validation**: Club names should have a minimum length (e.g., 3 characters) and maximum length (e.g., 100 characters)
4. **Automatic Fields**: The `id` field is automatically generated by Firestore when a document is created

### 2. clubMembers Collection

The `clubMembers` collection stores information about users who are members of clubs and their assigned roles.

#### Collection Path
```
/clubMembers/{memberId}
```

#### Document Structure

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | String | Yes (auto) | Firestore document ID, automatically assigned |
| `userId` | String | Yes | UID of the user who is a member |
| `name` | String | Yes | Display name of the member |
| `email` | String | Yes | Email address of the member |
| `clubId` | String | Yes | Reference to the club document ID |
| `role` | String | Yes | Member's role in the club |

#### Business Rules

1. **User Association**: The `userId` must correspond to an authenticated Firebase user
2. **Club Reference**: The `clubId` must reference an existing club document
3. **Unique Membership**: A user can only be a member of the same club once (unique combination of `userId` and `clubId`)
4. **Role Validation**: Valid roles could include: "Presidente", "Entrenador Principal", "Entrenador Asistente", "Manager", "Delegado", "Socio"
5. **Email Validation**: Email addresses should follow standard email format validation

## Relationships

### Club to ClubMembers (One-to-Many)

- One club can have multiple members
- Members are linked to clubs via the `clubId` field
- Query pattern: `clubMembers.where("clubId", "==", "<clubId>")`

### User to Clubs (One-to-Many as Owner)

- A user can own multiple clubs
- Ownership is tracked via the `ownerId` field in clubs
- Query pattern: `clubs.where("ownerId", "==", "<userId>")`

### User to ClubMembers (One-to-Many)

- A user can be a member of multiple clubs
- Membership is tracked via the `userId` field in clubMembers
- Query pattern: `clubMembers.where("userId", "==", "<userId>")`

## Data Access Patterns

### Common Queries

1. **Get all clubs owned by a user**
   ```javascript
   db.collection('clubs').where('ownerId', '==', userId).get()
   ```

2. **Get all members of a club**
   ```javascript
   db.collection('clubMembers').where('clubId', '==', clubId).get()
   ```

3. **Get all clubs where a user is a member**
   ```javascript
   db.collection('clubMembers').where('userId', '==', userId).get()
   // Then fetch club details for each clubId
   ```

4. **Find club by invitation code**
   ```javascript
   db.collection('clubs').where('invitationCode', '==', code).get()
   ```

5. **Check if user is member of a specific club**
   ```javascript
   db.collection('clubMembers')
     .where('userId', '==', userId)
     .where('clubId', '==', clubId)
     .get()
   ```

## Security Considerations

### Firestore Security Rules

The following security rules should be implemented:

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Clubs collection rules
    match /clubs/{clubId} {
      // Allow read if user is the owner or a member of the club
      allow read: if request.auth != null && (
        resource.data.ownerId == request.auth.uid ||
        exists(/databases/$(database)/documents/clubMembers/$(request.auth.uid + '_' + clubId))
      );
      
      // Allow create if user is authenticated
      allow create: if request.auth != null && 
        request.resource.data.ownerId == request.auth.uid;
      
      // Allow update only by the owner
      allow update: if request.auth != null && 
        resource.data.ownerId == request.auth.uid;
      
      // Allow delete only by the owner
      allow delete: if request.auth != null && 
        resource.data.ownerId == request.auth.uid;
    }
    
    // ClubMembers collection rules
    match /clubMembers/{memberId} {
      // Allow read if user is the member or the club owner
      allow read: if request.auth != null && (
        resource.data.userId == request.auth.uid ||
        get(/databases/$(database)/documents/clubs/$(resource.data.clubId)).data.ownerId == request.auth.uid
      );
      
      // Allow create only by club owner
      allow create: if request.auth != null && 
        get(/databases/$(database)/documents/clubs/$(request.resource.data.clubId)).data.ownerId == request.auth.uid;
      
      // Allow update only by club owner
      allow update: if request.auth != null && 
        get(/databases/$(database)/documents/clubs/$(resource.data.clubId)).data.ownerId == request.auth.uid;
      
      // Allow delete by club owner or the member themselves
      allow delete: if request.auth != null && (
        get(/databases/$(database)/documents/clubs/$(resource.data.clubId)).data.ownerId == request.auth.uid ||
        resource.data.userId == request.auth.uid
      );
    }
  }
}
```

## Indexes

### Recommended Composite Indexes

1. **clubMembers by clubId and role**
   - Collection: `clubMembers`
   - Fields: `clubId` (Ascending), `role` (Ascending)
   - Use case: Query members by club and filter by role

2. **clubMembers by userId and clubId**
   - Collection: `clubMembers`
   - Fields: `userId` (Ascending), `clubId` (Ascending)
   - Use case: Check membership and prevent duplicates
