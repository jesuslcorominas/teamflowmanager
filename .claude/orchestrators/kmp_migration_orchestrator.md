# KMP Migration Autonomous Orchestrator

You are operating in FULL AUTONOMOUS MODE.

Your mission is to execute the KMP migration described in `KMP_MIGRATION_PLAN.md`.

## Global Rules

- Do NOT ask for approval between steps.
- Do NOT stop unless:
    - A blocking technical error occurs
    - Tests fail repeatedly and cannot be fixed
    - Explicit user intervention is required
- You may create branches, commits, PRs and GitHub issues.
- You must self-review every PR before marking it ready.
- The user approval is only required when:
    - A migration milestone is complete
    - Or architectural decisions exceed the migration scope

---

## Phase 0 — Setup

1. Create base branch:

```bash
git checkout -b migration/kmp-migration
git push -u origin migration/kmp-migration
```

2. Parse KMP_MIGRATION_PLAN.md

3. Extract:

- Phases

- Milestones

- Independent tasks

- Dependencies between tasks

## Phase 1 — Issue Creation

For each atomic migration task:

1. Create a GitHub issue

- Label it:

    - kmp

    - migration

    - autogenerated

- Include:

    - Context

    - Acceptance Criteria

    - Technical constraints

    - Testing requirements

    - Definition of Done

## Phase 2 — Implementation Loop

For each issue:

1. Create branch:

```bash
migration/kmp-{issue-number}-{short-name}
```

2. Assign to Android Senior Dev Agent

3. Implement

4. Run tests

5. Ensure coverage does not decrease

6. Open PR targeting migration/kmp-migration

7. Assign PR Review Agent

## Phase 3 — PR Review Loop

PR Reviewer must validate:

- Code style

- Architecture

- KMP compatibility

- Test quality

- Coverage

- Dead code

- Unnecessary Android coupling

- Public API stability

If issues found:

- Comment

- Fix in same branch

- Re-run review

Only when PR is clean:

- Mark Ready for Review

- Add label: ready-for-human-approval

## Phase 4 — Merge Policy

- PRs must NOT auto-merge.

- Wait for human approval.

- After approval → merge into migration/kmp-migration.

Execution Mode

Continue automatically with next issue after merge.

Never stop between issues.

Stop only when PHASE 1 is completed.
